# Задание: Сервис проверки статуса сертификатов

## Контекст

Удостоверяющий центр (CA) выпускает миллионы сертификатов. Вам нужно создать простой, но продуманный сервис на Go, который отвечает на вопрос: **действителен ли сертификат с данным серийным номером на указанный момент времени** (или на текущий момент). Сервис будет использоваться как внутренний компонент для аудита и валидации.

## Что нужно сделать

### Часть 1. Реализация базового функционала (код)

Напишите HTTP-сервис с одним эндпоинтом:

**`GET /api/v1/check`** — принимает параметры запроса (query):

- `serial` (string, **обязательный**) — серийный номер в hex (например, `"01A2B3"`).
- `at` (string, *опциональный*) — время в формате RFC3339. Если не указан, используется текущее время сервера.

**Ответ:** JSON вида

```json
{
  "serial": "01A2B3",
  "valid": true,
  "reason": ""
}
```

или в случае невалидности:

```json
{
  "serial": "01A2B3",
  "valid": false,
  "reason": "expired|revoked|not_found"
}
```

#### Требования

- **Хранилище:** in-memory (например, `map[string]Certificate` или `sync.Map`). Заполните его несколькими тестовыми записями при старте (можно захардкодить).
- **Структура сертификата:**

  ```go
  type Certificate struct {
      Serial    string
      NotBefore time.Time
      NotAfter  time.Time
      RevokedAt *time.Time // nil, если не отозван
  }
  ```

- **Логика проверки:**
  1. Проверить наличие серийного номера. Если нет → `not_found`.
  2. Определить целевую дату `checkTime` (из параметра `at` или `time.Now()`).
  3. Убедиться, что `checkTime` попадает в `[NotBefore, NotAfter]`. Если нет → `expired`.
  4. Если сертификат отозван (`RevokedAt != nil`) и `checkTime >= RevokedAt` → `revoked`.
  5. Иначе → `valid: true`.

- **Валидация входных данных:** проверьте, что `serial` не пустой и состоит из hex-символов; `at` (если передан) должен быть корректным RFC3339. При ошибках возвращайте `400` с описанием.
- **Graceful shutdown** реализовывать не обязательно, но код должен быть готов к его добавлению (используйте `context.Context` в хендлере).

**Ожидаемый результат:** рабочий сервис, который можно запустить и проверить через curl/postman. Код должен быть читаемым, разделённым на логические пакеты/файлы (хотя бы `handler`, `storage`, `model`).

### Часть 2. Архитектурный эскиз (текст)

Представьте, что сервис работает со **100 УЦ**, каждый из которых выпускает сертификаты. Сервис должен выдерживать **10 000 запросов в секунду**, а база данных содержит **100 млн записей**. Кратко (не более 1 страницы) опишите, как бы вы изменили архитектуру для production.

Осветите следующие вопросы:

- **Хранение:** какую СУБД выберете и почему? Как будете индексировать/партиционировать данные для быстрого поиска по серийному номеру?
- **Кэширование:** что и где будете кэшировать для снижения нагрузки на БД? Как инвалидировать кэш при обновлении статусов?
- **Масштабирование:** как организуете горизонтальное масштабирование сервиса и базы данных?
- **Надёжность:** как сервис будет узнавать о новых сертификатах и отзывах?
