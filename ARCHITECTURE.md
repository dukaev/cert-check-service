# Архитектурный эскиз и тестируемость

Документ покрывает Часть 2 задания (production-архитектура для 100 УЦ, 10k RPS, 100 млн записей) и подходы к её проверке тестами + архитектурные швы, которые делают код пластичным.

---

## Часть 2. Production-архитектура

### 1. Хранение

**Выбор СУБД: PostgreSQL** (как основной кандидат), либо ScyllaDB/Cassandra при экстремальных требованиях к записи.

**Почему PostgreSQL:**
- Данные структурированные и почти неизменяемые (после выпуска меняется только `revoked_at`) — реляционная модель ложится естественно.
- 100 млн строк — нагрузка, с которой PG уверенно справляется на одном узле (типичная строка ~100 байт → ~10–20 ГБ; влезает в память).
- Доступ — точечный по PK (`serial`), это самая дешёвая операция в B-tree.
- Зрелая экосистема: репликация, partitioning, point-in-time recovery, Patroni.
- Транзакции нужны при публикации CRL — Postgres даёт ACID без дополнительной возни.

**Схема:**

```sql
CREATE TABLE certificates (
    ca_id        SMALLINT     NOT NULL,
    serial       BYTEA        NOT NULL,   -- hex храним как bytea, не строку
    not_before   TIMESTAMPTZ  NOT NULL,
    not_after    TIMESTAMPTZ  NOT NULL,
    revoked_at   TIMESTAMPTZ,
    PRIMARY KEY (ca_id, serial)
) PARTITION BY HASH (ca_id);
```

**Индексация/партиционирование:**
- **PK `(ca_id, serial)`** — серийник уникален только в пределах УЦ; запросы практически всегда приходят с указанием УЦ (либо его можно вывести из контекста клиента/маршрута).
- **HASH-партиционирование по `ca_id`** на 32–64 партиции — равномерно размазывает 100 УЦ и позволяет масштабировать запись/вакуум. Range по дате тут не помогает: запрос идёт по серийнику, а не по диапазону времени.
- **Покрывающий индекс** `(ca_id, serial) INCLUDE (not_before, not_after, revoked_at)` — index-only scan, без захода в heap.
- **`serial BYTEA` вместо TEXT** — меньше байт, быстрее сравнение, нет проблем с регистром hex.
- Если запросов «только по serial без ca_id» много — добавить вторичный индекс по `serial`, но лучше пробрасывать `ca_id` с клиента (issuer известен из самого сертификата).

### 2. Кэширование

**Что кэшировать:**

| Слой | Что | Где | TTL |
|---|---|---|---|
| L1 | Горячие `(ca_id, serial) → status` | In-process LRU/Ristretto в каждом инстансе | 30–60 с |
| L2 | То же + «недавно проверенные» | Redis (cluster) | 5–10 мин |
| Negative cache | `not_found` | L1+L2 | 1–2 мин (короткий, чтобы новый выпуск быстро «проявился») |
| CRL / список отзывов | компактный набор `revoked_serials` per ca_id | Redis + in-memory bloom filter | до следующего CRL update |

**Bloom filter по отозванным серийникам** на каждом инстансе — решает 99% запросов без обращения к Redis/БД: если фильтр говорит «не отозван» — это гарантия; если «может быть отозван» — идём в БД. Размер на 1 млн отозванных ≈ 1.2 МБ при FP 1%.

**Инвалидация:**
- Записи о сертификатах **иммутабельны до отзыва** — кэшировать `valid` ответы можно агрессивно, главное — корректно обрабатывать revocation.
- При отзыве: publisher пишет в БД и **публикует событие в Kafka/NATS** `cert.revoked` с `(ca_id, serial)`.
- Сервис подписан на этот топик; на каждом инстансе хендлер делает: `cache.Delete(key)` + `bloomFilter.Add(serial)` + `redis.DEL`.
- Дополнительно — **версия CRL** (`crl_version` per ca_id) в Redis: ключ кэша = `{ca_id}:{serial}:{crl_version}`, при бампе версии старые ключи естественно протухают.
- TTL на L2 короткий, чтобы при потере события событийная инвалидация подстраховывалась временем.

### 3. Масштабирование

**Сервис (stateless):**
- Горизонтально — N инстансов за L7-балансировщиком (nginx/envoy/cloud LB).
- Один pod при разумной обработке хендлера держит 2–5k RPS на ядро → 10k RPS укладывается в 4–8 подов с запасом.
- Автоскейлинг по CPU и p95 latency (HPA в k8s).
- **Sharding-aware роутинг:** балансировщик может хешировать по `ca_id` → инстанс, тогда in-memory bloom-фильтры становятся горячими по конкретным УЦ и cache hit-rate растёт.

**База:**
- **Read replicas** (3–5 штук) — все `/check` идут на реплики; запись (новые сертификаты, revocations) — на primary. С учётом кэша к реплике приходит лишь cache-miss трафик (1–5% от 10k = 100–500 RPS) — это не проблема даже для одной реплики.
- **HASH-партиции** уже подготовили почву; при росте до миллиардов — переход на шардинг (Citus/Vitess аналог, либо приложение-уровневый шардинг по `ca_id mod N`).
- **Connection pooling** через PgBouncer (transaction mode) — критично при большом числе подов.

### 4. Надёжность (как узнаём о новых сертификатах и отзывах)

Два источника правды и два потока:

**a) Inbound от УЦ → publisher-сервис → шина событий → consumer → БД:**
- Каждый УЦ публикует события (issuance / revocation) через:
  - вебхук в наш API (`POST /internal/ca/{ca_id}/events`) с HMAC-подписью, либо
  - выгрузку CRL/delta-CRL по расписанию (fallback / периодическая сверка).
- События складываются в **Kafka** (`cert.issued`, `cert.revoked`) — гарантия at-least-once, ordering per `ca_id` через ключ партиции.
- Consumer-воркер пишет в Postgres (idempotent UPSERT по `(ca_id, serial)`), фиксирует offset, публикует cache-invalidation event.

**b) Периодическая сверка (eventual consistency safety net):**
- Раз в N минут — pull CRL с каждого УЦ, сравнение с БД, добор разошедшегося. Защищает от потерянных вебхуков.

**Гарантии:**
- **Idempotent writes:** `INSERT ... ON CONFLICT (ca_id, serial) DO UPDATE SET revoked_at = EXCLUDED.revoked_at WHERE certificates.revoked_at IS NULL` — повторная доставка события не ломает данные.
- **Outbox pattern** в publisher: запись в БД и событие в Kafka — в одной транзакции через outbox-таблицу, чтобы не потерять revocation при падении.
- **Мониторинг рассинхрона:** метрика `crl_lag_seconds` per ca_id; алерт, если события не приходят > X минут.
- **Идемпотентный consumer** + DLQ для битых событий.
- Health-чеки сервиса проверяют доступность БД (read), Redis и наличие свежих сообщений в Kafka.

## Архитектурные швы для пластичности кода

Каждая граница из эскиза выше — это **интерфейс**, чтобы менять реализацию без переписывания вызывающего кода.

### 1. `Store` — узкий интерфейс, расширяемый без поломки контракта
Сейчас в `internal/storage/storage.go` уже есть `Get(serial)`. Эволюционный шаг:

```go
type Store interface {
    Get(ctx context.Context, caID uint16, serial []byte) (model.Certificate, error)
}
```

- `context.Context` — для таймаутов, трейсинга, отмены запросов (Postgres-реализация без него не живёт).
- `caID` — даже если сегодня всегда `0`, домен сразу честный: серийник уникален в пределах УЦ.
- `serial []byte` — чтобы при переходе на Postgres не делать конверсию hex↔bytea в двух местах.
- Ошибки **типизированные**: `ErrNotFound`, `ErrUnavailable` — handler сам решит, что вернуть клиенту.

### 2. `Cache` как декоратор над `Store`
Не отдельный сервис, а **обёртка с тем же интерфейсом**:

```go
type CachedStore struct { inner Store; cache *ristretto.Cache; bloom *bloom.Filter }
func (c *CachedStore) Get(...) // та же сигнатура
```

In-memory сборка и production-сборка отличаются только конструктором — вся остальная логика идентична. Кэш-инвалидацию выносим в отдельный интерфейс `Invalidator`, чтобы обработчик событий не знал про Ristretto.

### 3. `Clock` — инъекция времени
Сейчас в чекере время приходит параметром — это **уже правильно**. Но в handler `time.Now()` вызывается напрямую — это надо обернуть:

```go
type Clock interface { Now() time.Time }
```

Без этого нельзя нормально тестировать «при `at` не указан, берём текущее время» и сценарии «сертификат истёк секунду назад».

### 4. `EventSource` — абстракция источника обновлений
Kafka consumer, HTTP webhook или периодический CRL puller — это всё одно:

```go
type Event struct { Kind EventKind; Cert model.Certificate; OccurredAt time.Time }
type EventSource interface { Subscribe(ctx context.Context, handler func(Event) error) error }
```

Сегодня — заглушка, читающая из канала. Завтра — Kafka. Логика применения событий к Store + Cache одна и та же.

### 5. Идемпотентность с самого начала
Даже в in-memory сторе делать `Upsert`, а не `Insert`/`Update`. При переезде на Kafka at-least-once это уже встроено в код и не потребует переделки.

### 6. Чистая функция `Check`
В `internal/checker/checker.go` именно так и сделано (`cert + time → (valid, reason)`). Это **самый важный шов**: он позволяет property-based тестировать всю бизнес-логику без I/O.

### 7. Типизированные доменные ошибки и `reason`
`reason` (`expired|revoked|not_found`) — это публичный контракт, его нельзя хардкодить как строку в трёх местах. Константы уже вынесены в `checker.go`.

### 8. Метрики/трейсы как интерфейс
`MetricsRecorder` с no-op реализацией в dev — потом подменяется на Prometheus/OTel без правок бизнес-кода.

## Тесты — по уровням пирамиды

### 1. Unit (pure, быстрые, без I/O)

**`checker.Check` — table-driven по всем граничным случаям:**
- `at == NotBefore` (inclusive — valid)
- `at == NotAfter` (inclusive — valid)
- `at == NotAfter + 1ns` (expired)
- `at == RevokedAt` (revoked, по условию `>=`)
- `at == RevokedAt - 1ns` (valid)
- `at < NotBefore` (expired — «приоритет окна над отзывом» — это правило задания, его обязательно зафиксировать тестом)
- `at > NotAfter && RevokedAt != nil && at >= RevokedAt` — expired (по приоритету)

**Handler — валидация входа:**
- пустой `serial`, не-hex символы, очень длинный `serial`, регистр (`a` vs `A`)
- невалидный `at` (не RFC3339), будущий `at`, `at` без таймзоны
- отсутствующий `at` → используется `Clock.Now()` (проверяется через mock-часы)

### 2. Property-based (на чистой функции `Check`)
Через `testing/quick` или `gopter`:
- **Инвариант 1**: если `at ∈ [NotBefore, NotAfter]` и `RevokedAt == nil` → всегда valid.
- **Инвариант 2**: если `at > NotAfter` → всегда expired (независимо от `RevokedAt`).
- **Инвариант 3**: если valid → выставление `RevokedAt = at` сделает результат revoked.
- **Идемпотентность**: `Check(c, t) == Check(c, t)` (детерминизм).

Даёт уверенность, что любые будущие правки логики не сломают семантику.

### 3. Contract-тесты на `Store`
Один и тот же набор тестов прогоняется против **любой** реализации Store (in-memory, Postgres, Postgres+cache):

```go
func RunStoreContract(t *testing.T, factory func() Store) { /* общие сценарии */ }
```

Гарантия, что замена in-memory на Postgres не изменит наблюдаемое поведение. Контракт включает: «отсутствующий serial возвращает ErrNotFound», «Upsert revoked_at идемпотентен», «параллельные Get безопасны», и т. д.

### 4. Integration (testcontainers)

**Postgres:**
- Поднимается реальный Postgres, накатывается схема с партиционированием.
- Заливается синтетический датасет (хотя бы 1M записей, разбросанных по `ca_id`).
- Тест-кейс: `EXPLAIN ANALYZE` проверяет, что план — **index-only scan + partition pruning**, а не seq scan. Это **тест на архитектурное решение**, а не на логику.
- p99 latency lookup < N ms на горячем датасете.

**Redis + сервис:**
- Cache miss → запрос дошёл до Postgres → ответ закэширован → следующий запрос не дёргает Postgres (проверяем через мок/счётчик).

**Kafka:**
- Публикуем `cert.revoked` дважды → в БД один эффект, в кэше ключ инвалидирован один раз (или несколько — но идемпотентно).
- Падение consumer посередине → после рестарта от последнего committed offset событие обработано (at-least-once).

### 5. Тесты на надёжность / расхождение

- **Outbox:** эмулируем падение между записью в БД и публикацией события — после восстановления outbox-воркер дошлёт событие. Тест проверяет, что revocation не теряется.
- **Reconciliation:** руками портим состояние (отзываем серт в БД мимо событийной шины) → периодическая сверка с CRL обнаруживает расхождение и публикует cache-invalidation.
- **Replica lag:** пишем в primary, сразу читаем — read-your-writes для publisher-сервиса (через `LSN`/`pg_wait_for_lsn` или sticky reads на primary).

### 6. Chaos / degradation
- **Redis недоступен** → сервис отвечает, но из БД (фиксируем рост latency, но не 5xx).
- **Postgres недоступен**, кэш горячий → отдаём из кэша с пометкой `stale=true` (если допустимо политикой).
- **Kafka отстаёт** → метрика `event_lag_seconds` растёт, но `/check` работает.

Тесты не на «корректность», а на «архитектура держит то, что обещано».

### 7. Performance / benchmarks
В проекте уже есть `checker_bench_test.go` и `storage_bench_test.go` — правильный задел. Расширить:
- **Микробенч** `Check` — должен быть ~единицы наносекунд (потолок пропускной способности на ядро).
- **Бенч in-memory Store** — нижняя граница хранилища.
- **Load test** (vegeta/k6) против полного сервиса с Postgres+Redis — целевые SLO: 10k RPS, p99 < 50 ms, error rate < 0.01%. Это **прямая верификация** заявленной цифры из Части 2.
- **Soak test** на 1–2 часа — ловим утечки и деградацию hit-rate кэша.

### 8. Fuzz
`go test -fuzz` на парсер `serial` и `at`. Hex-парсинг и парсинг времени — классические места, где fuzz находит UB.

## Сводка: какое решение чем проверяется

| Архитектурное решение из Части 2 | Чем проверяется |
|---|---|
| Partition pruning + index-only scan в Postgres | Integration-тест с `EXPLAIN` |
| Кэш не отдаёт устаревший статус | Integration: revoke → инвалидация → следующий запрос видит revoked |
| At-least-once consumer идемпотентен | Integration: два одинаковых события → один эффект |
| Сервис горизонтально масштабируется | Load-тест против N подов; stateless проверяется тем, что rolling-restart не теряет запросы |
| Деградация без Redis/Kafka | Chaos-тест с убитой зависимостью |
| Бизнес-логика корректна на границах | Unit + property-based на `Check` |
| Любая реализация Store эквивалентна | Contract-тест прогоняется на всех |

---

**Главный принцип:** бизнес-логика (`Check`) — чистая и покрыта property-based; всё, что снаружи — за интерфейсами, и для каждого интерфейса есть контракт-тест. Переезд in-memory → Postgres+Redis+Kafka превращается не в переписывание сервиса, а в подключение новых реализаций к уже протестированным контрактам.
